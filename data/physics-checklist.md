# 3Dからくり図鑑 — 物理法則・機構チェックリスト

> **このドキュメントは、各エピソードの3Dモデルを作成・レビューする際に、
> 毎回必ず確認すべき項目をまとめたものです。**
>
> EP02（ラック＆ピニオン機構）の開発で得た知見をベースにしています。

---

## 1. 運動変換の正確性

すべてのからくりは「ある運動を別の運動に変換する」仕組み。
**入力 → 出力の変換公式が物理法則に合っていること**が最優先。

### チェック項目

- [ ] **変換公式を明記**する（コード内コメントに必ず書く）
  - カム機構: `y = amplitude × sin(waves × θ)`
  - ラック＆ピニオン: `s = r × θ`（移動量 = 半径 × 回転角）
  - 歯車: `ω₂ = ω₁ × (Z₁ / Z₂)`（角速度比 = 歯数の逆比）
  - リンク機構: 逆運動学で関節角度を算出
- [ ] **単位の整合性**：角度はラジアンで統一（Three.jsの仕様）
- [ ] **方向の整合性**：回転方向と移動方向の因果関係が正しいか
  - 例）反時計回り → 接触面が左へ → 接触部品も左へ

## 2. 歯車・歯のかみ合い

歯車を使う機構では「歯のかみ合い」が視覚的に最重要。
**子供が「歯と歯がかみ合って動いている！」と直感的に理解できること**が目標。

### チェック項目

- [ ] **ピッチ円が接する配置**
  - ピニオンのピッチ円半径 + ラック/相手歯車のピッチ円半径 = 中心間距離
  - これがずれると歯が浮いたり食い込んだりする
- [ ] **歯のピッチ（間隔）を統一**
  - ピニオンの歯ピッチ = `2π × ピッチ円半径 / 歯数`
  - かみ合う相手（ラック・歯車）の歯ピッチを同じ値にする
- [ ] **歯がかみ合う（インターロック）**
  - ピニオンの歯先が、ラック/相手歯車の歯溝に入り込む
  - 歯先円半径（outerR）が相手の歯底円を超える配置にする
  - 単に「隣接」ではなく「嚙み込む」表現にする
- [ ] **バックラッシュ（遊び）の表現**
  - 実物には微小な隙間（バックラッシュ）がある
  - 0.5〜1.0程度の隙間を設けると自然に見える
- [ ] **歯のプロファイル（断面形状）**
  - 台形歯：歯元が太く、歯先が細い（最低限これを守る）
  - より正確にはインボリュート曲線だが、簡易的な台形で十分
  - 参考：歯元幅 ≒ ピッチの50〜60%、歯先幅 ≒ ピッチの30〜40%

## 3. 部品の物理的制約

「現実ではありえない動き」をしていないか。
**画面の前の子供が「実物と違う！」と感じないこと。**

### チェック項目

- [ ] **部品同士が貫通しない**
  - ドア同士が交差しない、ギアが壁を突き抜けないなど
  - `Math.max()` / `Math.min()` でクランプする
- [ ] **可動範囲に限界がある**
  - ドアは閉じた位置で止まる（それ以上閉じない）
  - 傘は完全に開いた状態以上に広がらない
  - レバーには最大角度がある
- [ ] **重力方向を意識する**
  - 支えのない部品は落ちるべき位置にある
  - バネやストッパーがない限り、部品が浮いて見えないように
- [ ] **接続関係が明確**
  - 部品Aが部品Bを動かすとき、物理的な接続（軸・歯・リンク等）を表示
  - 「なぜ一緒に動くのか」が視覚的にわかる接続バーやシャフトを入れる

## 4. アニメーションのリアリティ

動き方が「機械らしい」こと。

### チェック項目

- [ ] **加減速をつける**
  - 突然動き始めて突然止まるのは不自然
  - smoothstep: `t² × (3 - 2t)` を使うとなめらかな開始・停止になる
  - ease-in-out: `sin()` ベースのカーブも有効
- [ ] **速度の因果関係**
  - 歯車のギア比が変わると、出力側の速度が変わる
  - 速度比 = ギア比の逆数
- [ ] **周期的な動きにはsin/cos**
  - カム機構の上下運動、クランクの往復運動など
  - `sin(ωt)` で位置、`cos(ωt)` で速度が導出できる
- [ ] **停止位置がある動きには状態遷移**
  - 開→停止→閉→停止 のようなサイクル
  - sin波だけではなく、ステート管理も検討する

## 5. 視覚的なわかりやすさ

子供が「構造を理解できる」ための表現上の工夫。

### チェック項目

- [ ] **パーツごとに色分け**
  - 機構のキーパーツは目立つ色（オレンジ、紫など）
  - フレームや筐体は控えめな色（グレー系）
  - 部品の機能が同じなら同系色で統一
- [ ] **重要な機構部分が見える**
  - 筐体やカバーは半透明 or ワイヤーフレーム表示
  - カメラ初期位置を「機構が一番見やすい角度」に設定
- [ ] **パーツ名のラベルを表示**
  - 3Dスプライトで「ピニオン」「ラック」等のテキストを表示
  - ラベルは部品と一緒に動く（追従する）
- [ ] **操作ヒントを明確に**
  - ドラッグで回転、スクロールでズーム等のヒントテキスト
  - 手動/自動の切り替えボタン

## 6. コード品質（再利用性）

図鑑を量産するためのコード設計。

### チェック項目

- [ ] **パラメーターを冒頭にまとめる**
  - マジックナンバーを避け、名前付き定数で定義
  - 物理的な意味をコメントに記載
- [ ] **物理公式をコメントに明記**
  - 「なぜこの計算をしているのか」をコード内に書く
  - 例: `// 物理法則: s = r × θ → θ = s / r`
- [ ] **共通スタイルは style.css を使う**
  - エピソード固有のスタイルは `<style>` で追加OK
  - レイアウト・カード・ボタンは共通CSSを使う
- [ ] **HTMLの構造を統一**
  - ヘッダー → 3Dビューアー → 解説グリッド → 大人向けコラム → ナビゲーション
  - 新規エピソード作成時はこの順番を守る
